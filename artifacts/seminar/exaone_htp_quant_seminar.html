3.Tied Word Embedding

Tied Word Embedding = 입력 embedding과 출력 LM Head가 같은 weight를 공유하는 모델 구조. 추론 때 켜고 끄는 성능 옵션이 아니라, 학습 시 parameter 공유 여부를 정하는 설계다.

비유 — 입력 embedding은 "이미 아는 단어를 내부 언어로 번역하는 사전", LM Head는 "내부 생각이 vocabulary의 어떤 단어와 가장 잘 맞는지 전체 사전과 비교하는 과정". EXAONE/HF 구현에서는 두 모듈이 별도로 존재하더라도 tie_word_embeddings=true이면 weight Parameter가 같은 메모리를 공유하도록 구현될 수 있다.

이 모델은 Tied로 확인됨 확정 — 이 저장소의 EXAONE GGUF에는 output.weight/lm_head.weight tensor가 존재하지 않고, backend 로그에서 token_embd.weight가 MUL_MAT의 입력으로 vocab logits(2048×65536→65536)를 생성합니다. 또한 llama.cpp 양자화기는 has_tied_embeddings = true를 기본 가정합니다(output.weight를 만나기 전까지). 근거: 13장.

이 화면 한 문장: "이 모델은 Tied입니다 — 그래서 embedding 하나의 backend 문제가 입력과 출력 양쪽에 동시에 영향을 줍니다."
단정 금지: tied 여부는 "추측"이 아니라 tensor 목록·로그로 확인한 것이라고 강조. 다른 모델은 다를 수 있음.
연결: "그럼 이 token_embd가 HTP에서 지원되는지가 중요해집니다(9장)."

4.Prefill과 Decode

Prefill — 입력 255 tokens 병렬

입력 token 여러 개를 한꺼번에 처리. 각 position의 K/V를 계산해 KV cache에 저장한다.

직전 token 1개 입력 → 다음 token 1개 생성을 반복. 과거 K/V는 다시 계산하지 않고 cache를 참조·append.

  • Decode는 과거 token의 K/V를 다시 계산하지 않는다. 하지만 새 token 하나가 모든 30개 block을 통과한다.
  • Attention은 과거 KV cache 전체를 읽으므로 context가 길수록 메모리 접근량이 증가한다.
  • HTP는 큰 병렬 연산엔 유리하지만, batch 1과 잦은 backend 전환에는 불리할 수 있다 가설.
이 화면 한 문장: "Prefill은 한 번의 큰 연산, Decode는 작은 연산의 무한 반복 — 그래서 최적 backend가 다를 수 있습니다."
질문 예상: "KV cache 없으면?" → 매 step prompt 전체 재계산. cache가 decode를 가능하게 함.
연결: "이 batch-1 반복이 뒤 Decode 성능 해석의 핵심입니다."

5.양자화 개념

양자화 = weight를 더 적은 비트로 저장. 타입별로 backend 지원성이 다른 것이 이 프로젝트의 핵심.

타입bpw구조HTP 정황(이번 분석)
F161616-bit float, 품질 기준점HTP에 대형 weight 배치됨
Q8_0~8.5GGUF block quant, ~8-bit weight (W8A8 아님)HTP weight offload 정황
Q4_0~4.532 weight당 4-bit+scaleHTP offload 확인 — 현실적 PoC 후보
Q4_K~4.5256 super-block + 하위 block scale/min (계층형)HTP에 대형 weight buffer 미생성 정황
Q8_0 주의 — GGUF Q8_0은 weight를 약 8-bit로 저장하는 weight block quantization이며, activation까지 int8로 만드는 W8A8 activation quantization과 동일 개념이 아닙니다.

Q4_K_S / Q4_K_M / Q4_K_pure — recipe 구분

storage type vs recipe — Q4_K는 개별 tensor 저장 타입이고, Q4_K_S·Q4_K_M은 모델 전체 양자화 recipe(어떤 tensor를 어떤 타입으로 승격할지). Q4_K_pure는 가능한 tensor를 기본 Q4_K로 유지하는 실험 구성이다.

llama.cpp의 use_more_bits(i_layer, n_layers)가 "더 정밀하게 보호할 layer"를 고른다(근거: llama-quant.cpp:417). EXAONE 30-layer에서 선택된 layer:

use_more_bits layers = 0, 1, 2, 5, 8, 11, 14, 17, 20, 23, 26, 27, 28, 29 (14개)

Q4_K_S와 Q4_K_M의 실제 tensor 차이(저장소 GGUF 비교 결과, 총 29개): attn_v 15개 ffn_down 14개. M은 이들을 Q6_K로 승격한다. Tied로 추정되는 token_embd.weight는 S·M 모두 Q6_K였다.

8.메모리 로그 기반 해석

Q4_0 — HTP로 실제 이동한 강한 증거 확정

CPU-only 실행에서 CPU_REPACK ≈ 678.75 MiB. HTP0 실행에서는 그 대부분이 HTP layout으로 이동한다.

Q4_0 bufferCPU-onlyHTP0 실행
HTP0-REPACK—573.75 MiB
CPU_REPACK678.75 MiB105.00 MiB
계산 — 573.75 + 105.00 = 678.75 MiB (CPU-only와 동일). HTP repack 비율 = 573.75 / 678.75 ≈ 84.5%. Q4_0 주요 Transformer weight가 HTP용 layout으로 이동한 강한 증거다.
단, — 남은 CPU 약 105 MiB를 검증 없이 LM Head라고 확정하지 말 것. embedding/output 계열(Q6_K token_embd) 및 norm일 가능성이 있으나, tensor별 배치를 확인하기 전까지 미확정 확인 필요.

Q4_K_pure — HTP에 대형 weight 없음 확정

Q4_K_pure buffer (HTP0 실행)값
CPU model buffer72.00 MiB
CPU_REPACK645.75 MiB
HTP0 model buffer0.50 MiB
HTP0-REPACK없음/유의미하지 않음
해석 — Q4_K 대형 weight는 CPU에 남아 있다. HTP0 model buffer 0.50 MiB는 context/control buffer 수준일 가능성이며 모델 전체 offload로 표현하면 안 된다. 현재 Hexagon backend에서 K-quant weight kernel 미지원 정황.
이 화면 한 문장: "Q4_0은 84.5%가 HTP로 갔고, Q4_K는 0.5MiB만 갔습니다 — 같은 'htp_layers=31'인데 실제는 정반대입니다."
단정 금지: 105MiB=LM Head 확정 금지. 0.5MiB=offload 아님만 확정.

9.GET_ROWS 로그와 CPU↔HTP Graph 분할

Tied embedding의 token_embd.weight(Q6_K)가 HTP에서 지원되는지 backend capability 로그로 확인한다.

HTP0 supports-op GET_ROWS|token_embd.weight x inp_tokens -> embd|2048:65536 x 16:1 -> 2048:16|f16 x i32 -> f32|2:4096 x 4:64 -> 4:8192|CPU x NONE -> NONE|no
  • GET_ROWS(token_embd.weight)는 HTP0에서 지원되지 않음(no) 확정. 따라서 token embedding 연산은 CPU에 남을 가능성이 높다.
  • 저장소 로그에는 MUL_MAT(token_embd.weight → 65536 logits)도 no로 나타난다 — 즉 tied LM Head도 HTP capability 실패.
  • supports-op no는 backend capability 검사 결과이지 실제 실행 trace가 아니다. 실제 실행 backend는 scheduler assignment/execution trace로 추가 확인 필요 확인 필요.
두 표현을 구분 —
"GET_ROWS가 CPU에서 더 빠르다" → 현재 데이터로 증명되지 않음.
"GET_ROWS가 HTP 미지원이라 CPU graph에 남는다" → 로그로 확인된 사실.

Decode graph 분할 가설 가설

CPU — Token ID → GET_ROWS(token_embd) → embedding
▼ CPU → HTP boundary (dispatch/transfer)
HTP — Transformer block 0~29 (Q4_0 MUL_MAT = HTP supported) → final hidden
▼ HTP → CPU boundary 가능성
CPU 또는 HTP — Final RMSNorm → LM Head(tied, Q6_K MUL_MAT = HTP no) → logits → sampling
용어 — Graph=연산 연결 구조 · Subgraph=동일 backend 연속 구간 · Backend boundary=실행 주체 전환점 · Dispatch · Synchronization · Tensor transfer.

핵심 가설: Decode는 token 하나당 이 경로가 반복된다. Prefill은 큰 계산이라 HTP dispatch 비용을 상쇄하지만, Decode는 batch 1이라 고정 overhead 비중이 커진다. LM Head까지 CPU라면 token당 CPU→HTP→CPU 왕복이 생길 수 있다.

아직 확정하지 말 것 — ① 전체 Decode 저하가 GET_ROWS 하나 때문 ② LM Head가 반드시 CPU 실행 ③ CPU↔HTP copy가 전체 지연의 주원인 ④ HTP batch-1 kernel 자체가 느린지. 모두 추가 profiling 대상 확인 필요.
이 화면 한 문장: "embedding과 tied LM Head는 HTP가 지원하지 않아 CPU에 남고, 그 사이 Transformer만 HTP라 경계 왕복이 생길 수 있습니다 — 이건 아직 '가설'입니다."
단정 금지: supports-op(capability)를 실행 trace라고 말하지 말 것. "확인됨"과 "가설"을 색으로 구분.

11.핵심 결론 — 확인된 사실 vs 가설

✔ 확인된 사실 확정

  1. F16·Q8_0·Q4_0은 HTP에 대형 weight buffer가 생성된다.
  2. Q4_0은 CPU repack 중 약 573.75 MiB가 HTP0-REPACK으로 이동(84.5%).
  3. Q4_K 계열은 htp_layers=31이 표시돼도 대형 HTP weight buffer가 생성되지 않았다(0.50 MiB).
  4. Q4_0 Prefill은 HTP가 CPU보다 크게 빠르다(≈3.49×).
  5. 모든 quant에서 현재 Decode는 CPU-only가 더 빠르다.
  6. GET_ROWS(token_embd.weight)는 HTP0 미지원(no). 이 모델은 tied.

◐ 현재 가설 가설

  1. Decode graph가 CPU·HTP subgraph로 분할된다.
  2. token마다 dispatch·synchronization·tensor 전달이 반복된다.
  3. HTP의 batch-1 효율이 낮을 수 있다.
  4. tied LM Head가 CPU에 남아 있을 가능성.
  5. 위 요인이 복합적으로 Decode 지연을 만든다.

◆ 아직 필요한 검증 확인 필요

1. 각 graph node의 assigned/executed backend
2. Transformer block별 CPU/HTP 실행 여부
3. LM Head의 실제 backend
4. layer·op별 실행 시간
5. CPU→HTP / HTP→CPU transition 횟수
6. copy·dispatch·synchronize 시간
7. Q4_0·Q8_0의 layer별 CPU/HTP latency
8. cold TTFT vs warm TTFT 분리
9. CSV 파싱 지표 일관성
